★ 01

Hierarchical Keystroke Decoder — Google-funded research

Real-time keystroke prediction from 3D hand-tracking motion capture, with no keyboard. Hierarchical Finger → Row → Key architecture, dual-specialist fusion, and a 5-minute per-user calibration phrase. Currently around 70% LOSO accuracy — actively pushing it higher.

LightGBM CatBoost Hand kinematics Hierarchical Per-user calibration LOSO eval Sub-3ms inference
The setup

The user types in the air — there is no physical keyboard. A motion-capture rig tracks 21 hand markers (4 per finger + wrist anchors, ~200 Hz). The model has to predict which key was intended from the kinematics alone. Every architectural choice in the system is shaped by that constraint.

The architecture

The hand is a 21-keypoint graph; every joint is expressed in wrist-relative coordinates rather than absolute world position — the single fix that unblocked cross-user generalisation, after a long detour where held-out users were collapsing on row detection. From there a three-stage hierarchy runs Finger → Row → Key, with the Row classifier (Top / Home / Bottom) gating the final softmax so the model can no longer commit row-mismatch errors.

Dual-specialist fusion

The key classifier itself is two CatBoost specialists trained on different feature subsets: Specialist A on velocity / acceleration / jerk (the dynamics signal), Specialist B on wrist-relative coordinates and joint angles (the position signal). Their probabilities are fused geometrically with a tunable temperature so each specialist tempers the other's overconfidence.

Per-user calibration (the part nobody publishes)

A new typist provides a 5-minute calibration phrase, and four corrections are fitted in sequence: a posture baseline (per-hand Y/Z mean), a per-finger shift map capturing each finger's biomechanical bias, per-class centroids in the model's discriminative subspace, and a vector-scaling logit bias with adaptive per-class regularisation. The result saves to a small user_profile.pkl that the runtime loads alongside the global model.

Where it stands

~70% LOSO accuracy is honest but not the destination. The kinematic features still leak too much within-session signal that doesn't transfer across users. Current work: stronger augmentation, sequence priors, and a denser per-user adaptation loop. No gaze — this project is purely hand-kinematic.

Companion repo

3D_hand_angles_plotting — the vectorised biomechanical preprocessing pipeline that feeds it. Scroll down to section 04 for the live architecture walkthrough.

03

improved-gaze-typing-prediction

Two-stage BiLSTM pipeline predicting typed characters from gaze + hand-tracking, with leave-one-subject-out evaluation across 5 participants.

PyTorch BiLSTM LOOCV Gaze + hand Hierarchical
Approach

Stage one predicts a finger / kinematic cluster from gaze + motion features; stage two conditions on that to narrow the character distribution. Leave-one-subject-out cross-validation across all participants — the only honest test of generalisation when within-session noise dominates.

Lessons

That participant-level normalisation matters more than another model layer. Robust scaling (median + IQR) per participant brought LOSO accuracy from ~55% into the 70s — a single preprocessing decision worth more than weeks of architecture search.

04

3D_hand_angles_plotting — biomechanical preprocessing

Vectorised pipeline computing signed MCP abduction and PIP/DIP flexion from motion-capture data, with 3D keystroke-cluster visualisation.

Python NumPy Motion capture Biomechanics Visualisation
What it does

Takes raw motion-capture marker data of the hand and emits a clean kinematic feature set: signed MCP abduction, MCP/PIP/DIP flexion, and angular velocities — all computed vectorised over the entire trial. Then plots the resulting feature space as 3D clusters per keystroke target, so it's actually visible whether kinematic separation exists before any model is trained.

Why it matters

This is the preprocessing layer underneath the keystroke-decoder work. If the kinematic features don't separate visually, no amount of model depth downstream will save it.

05

Automatic Paper Evaluation — multi-phase agentic pipeline

Research collaboration with Dr Daniele Giunchi: an agentic system that ingests a research paper and produces a structured, multi-axis evaluation.

Anthropic SDK Multi-agent ReAct + Reflexion RAG Tool use
The thesis

Reviewing a paper is a multi-stage cognitive task — extract claims, locate evidence, audit methodology, judge novelty. Each stage benefits from different prompting, different retrieval, different stop conditions. So we model each as its own agent in a coordinated pipeline rather than asking one prompt to do everything.

Building blocks

Claim extraction with grounded retrieval over the paper. Evidence-locator agent with citation-resolution tools. Methodology auditor with structured checklists. Reflexion loop on contradictions between stages. The whole thing produces a reviewer-style report you could hand to a programme committee.

07

Graph-Based Email & Meeting Scheduling Assistant

Production-ready FastAPI service that reads inbound email, reasons over real Google Calendar availability, and books meetings — but only after you say yes.

FastAPI Pydantic Gmail API Google Calendar Pub/Sub OAuth2
Architecture

Six clearly separated layers: ingestion, extraction, calendar reasoning, decision engine, calendar integration, API. Gmail Push Notifications via Cloud Pub/Sub trigger a webhook; an LLM-powered parser extracts intent; a slot reasoner scores availability; a decision engine routes to one of four paths (SCHEDULE, SUGGEST, RESCHEDULE, NO_ACTION).

The deliberate choice

Approval-required-before-booking. Even when the system is confident, it emails me numbered slot options and waits for "1", "2", or "3" before actually creating the calendar event. Full automation is the wrong target here — the cost of a wrongly-booked meeting is much higher than the cost of one extra reply.

03c Inside the Pipeline

From hand markers to a predicted keypress — the actual stages.

Scroll through the pipeline. Each stage lights up as you pass it; the architecture diagram on the left re-routes its data flow to match. Code on the right is real — pulled directly from the training file.

▸ system architecture · live
▸ Stage 01 · Preprocessing

Wrist-relative coordinates — the single fix that unblocked cross-user generalisation.

Every fingertip and joint is expressed relative to the wrist midpoint, removing dependency on absolute world position. This was the root cause of the row-detection collapse in held-out users.

▸ Stage 02 · Finger classifier

LightGBM gates the finger first — narrows the search before any key prediction.

Top-35 features by gradient-boosted importance, balanced class weighting, multi-class softmax over Index / Middle / Ring / Little. Gets a clean ~80% on its own; the keys downstream rely on this being right.

▸ Stage 03 · Row classifier

Top / Home / Bottom — row consistency as a hard prior on the final key.

The row classifier explicitly draws on wrist-relative Y features and MCP flexion means. Its softmax output later multiplies into the per-key probability, so the model can no longer commit row-mismatch errors.

▸ Stage 04 · Dual-specialist key fusion

Two CatBoost specialists — dynamics + positions — fused geometrically.

Specialist A trains on velocity / acceleration / jerk features; Specialist B on wrist-relative coords + joint angles. Their probabilities are fused with a tunable temperature so each gates the other's overconfidence. Row + Finger output then gates the final key softmax.

▸ Stage 05 · Per-user calibration

Five minutes of typing → a personal model. Four corrections, fitted in sequence.

A posture baseline (per-hand Y/Z mean), a per-finger shift map (each finger's biomechanical bias), per-class centroids in the model's discriminative subspace, and a vector-scaling logit bias with adaptive per-class regularisation that scales with how many calibration samples each class actually has.
